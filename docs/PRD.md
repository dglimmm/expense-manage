# 📋 PRD: 개인 비용 관리 사이트

## 1. 프로젝트 개요

| 항목          | 내용                                                                                                                                  |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| 프로젝트명    | 개인 비용 관리 사이트                                                                                                                 |
| 목적          | Notion을 CMS로 활용한 개인 비용 관리 사이트 구축                                                                                      |
| CMS 선택 이유 | Notion 데이터베이스에 글(지출 항목)을 작성하면 자동으로 사이트에 반영되어, 별도의 관리자 페이지나 백엔드 없이 콘텐츠를 관리할 수 있음 |

## 2. 주요 기능

1. **Notion 데이터베이스 연동**: Notion 데이터베이스에서 비용 목록을 가져와 사이트에 표시
2. **날짜별 필터링**: 특정 기간(연/월/일 단위)으로 지출 목록을 필터링
3. **엑셀 다운로드**: 필터링된 목록을 기존에 사용하던 엑셀 양식에 맞춰 다운로드
4. **반응형 디자인**: 모바일/태블릿/데스크톱 환경에서 모두 사용 가능한 UI

## 3. 기술 스택

| 영역       | 기술                                                                                  |
| ---------- | ------------------------------------------------------------------------------------- |
| Frontend   | Next.js 15, TypeScript                                                                |
| CMS        | Notion API (`@notionhq/client`)                                                       |
| Styling    | Tailwind CSS, shadcn/ui                                                               |
| Icons      | Lucide React                                                                          |
| 엑셀 생성  | `xlsx-populate` (기존 양식의 서식·수식·차트를 보존하며 셀 값만 채워 넣는 방식에 적합) |
| Deployment | Vercel                                                                                |

> **엑셀 생성 위치**: 클라이언트가 아닌 **서버(Route Handler, 예 `app/api/export/route.ts`)** 에서 워크북을 생성하여 바이너리로 응답합니다. 서버에서 처리하면 ① 무거운 엑셀 라이브러리가 클라이언트 번들에 포함되지 않고, ② 템플릿 파일을 파일시스템에서 직접 읽을 수 있습니다. 개인용 데이터 규모(수백 행)에서는 Vercel 함수의 응답 본문 제한(4.5MB)에 영향받지 않습니다.

## 4. Notion 데이터베이스 구조

| 필드명     | 타입   | 설명                 |
| ---------- | ------ | -------------------- |
| `content`  | Text   | 상세 내역            |
| `category` | Select | 사용 항목 (카테고리) |
| `useDate`  | Date   | 사용 날짜            |
| `cost`     | Number | 비용                 |

## 5. 화면 구성

| 화면        | 설명                                                    |
| ----------- | ------------------------------------------------------- |
| 홈          | 연간 지출 내역을 한눈에 보여주는 대시보드               |
| 날짜 필터링 | 사용자가 지정한 기간의 지출 목록을 조회/필터링하는 화면 |

## 6. MVP 범위

- Notion API 연동 (데이터베이스 조회)
- 비용 목록 조회 및 엑셀 다운로드
- 기본 스타일링 (Tailwind CSS + shadcn/ui)
- 반응형 디자인

## 7. 구현 단계

1. **환경 설정**: Notion API 패키지(`@notionhq/client`) 설치 및 프로젝트 환경 변수 설정
2. **Notion 연동 준비**: Notion 데이터베이스 생성 및 Integration API 키 발급/설정
3. **비용 목록 페이지 구현**: 날짜별 비용 목록을 조회하고 필터링하는 페이지 개발
4. **엑셀 다운로드 기능 구현**: 필터링된 비용 목록을 기존 엑셀 양식에 데이터를 채워 넣는 방식으로 다운로드 기능 개발
   1. `xlsx-populate` 라이브러리 설치
   2. 템플릿 파일(`src/lib/templates/expense-template.xlsx`)을 서버에서 로드
   3. 데이터 시작 셀 좌표(행/열) 및 컬럼-필드 매핑 정의 (아래 8절 참고)
   4. 필터링된 데이터를 셀에 기록하고 워크북을 버퍼로 생성
   5. `Content-Disposition` 헤더와 함께 파일 다운로드 응답 반환
5. **스타일링 및 최적화**: UI 다듬기, 반응형 대응, 성능 최적화

## 8. 엑셀 템플릿 관리

기존에 사용하던 엑셀 양식(서식·수식·레이아웃)을 그대로 유지한 채, 데이터만 채워 넣어 다운로드를 제공합니다.

### 8.1 보관 위치

- **레포 내 서버 전용 경로에 커밋**: `src/lib/templates/expense-template.xlsx`
  - 공개 노출이 필요 없는 파일이므로 `/public`이 아닌 서버 전용 디렉터리에 둡니다.
  - 외부 스토리지(S3, Vercel Blob 등)는 매 요청마다 네트워크 호출이 필요해 거의 변경되지 않는 개인용 고정 양식에는 과도한 설계이므로 채택하지 않습니다.

### 8.2 Vercel 배포 시 유의 사항

- Vercel 서버리스 함수는 빌드 시 함수 번들에 포함된 파일만 런타임에서 읽을 수 있습니다(파일시스템은 읽기 전용).
- Next.js가 `src/lib/templates/expense-template.xlsx` 같은 동적 파일 경로를 자동으로 추적하지 못할 수 있으므로, `next.config.ts`의 `outputFileTracingIncludes` 옵션으로 해당 파일을 함수 번들에 명시적으로 포함시켜야 합니다.

### 8.3 셀 매핑 정의 (코드 계약)

템플릿의 행/열 구조는 코드와의 계약이 되므로, 데이터 영역의 시작 좌표와 컬럼-필드 매핑을 문서화하고 양식 레이아웃 변경 시 함께 갱신합니다.

| 항목                   | 값 (예시 — 실제 양식에 맞게 확정 필요)               |
| ---------------------- | ---------------------------------------------------- |
| 데이터 시작 셀         | `A5`                                                 |
| 컬럼 매핑              | A: `useDate`, B: `category`, C: `content`, D: `cost` |
| 헤더/합계 등 정적 영역 | 데이터 기록 시 변경하지 않음                         |

## 9. 향후 고려 사항 (Out of Scope for MVP)

- 지출 항목 직접 추가/수정/삭제 기능 (Notion에서만 관리)
- 카테고리별 통계/시각화(차트)
- 다중 사용자 지원 및 인증
- Notion API 페이지네이션/rate limit 대응 고도화 (데이터 양 증가 시 `start_cursor` 기반 페이지네이션 적용 검토)
